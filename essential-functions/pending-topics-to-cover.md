## P0 - 最高优先级（缺失核心内容）

### 1. 文件系统沙箱与工作区隔离 (Worktree + Task Isolation)

**问题陈述**：
- 多个 Agent 任务并行执行时如何避免文件冲突？
- 如何隔离不同任务的文件系统视图？
- 临时工作区的生命周期如何管理？

**需要覆盖的技术点**：
- Git worktree 机制用于多任务并行
- 每个任务独立目录的创建与清理
- 目录级权限控制（读/写/执行边界）
- 工作区与任务的 ID 绑定策略
- 跨工作区文件访问的安全策略

**参考实现**：
- Claude Code 的 worktree 隔离模式
- Docker 容器内的 bind mount 策略
- 临时目录清理与垃圾回收

**关键设计问题**：
- 工作区复用 vs 每次都新建？
- 大文件跨工作区拷贝的性能问题
- 工作区失败后的残留清理

---

### 2. 子智能体架构 (Subagents)

**问题陈述**：
- 主 Agent 如何委派任务给子 Agent？
- 子 Agent 应该继承还是独立维护消息上下文？
- 子 Agent 的结果如何汇总回主流程？

**需要覆盖的技术点**：
- 子智能体的生命周期管理（创建/执行/销毁）
- 父-子消息传递机制（同步 vs 异步）
- 独立消息上下文 vs 共享上下文的权衡
- 子智能体结果聚合策略（顺序/并行/条件）
- 子 Agent 错误传播与父级恢复

**参考实现**：
- IronClaw 的 ChatDelegate/JobDelegate/ContainerDelegate 模式
- Edict 的子部门（六部）委派机制
- 多进程/多线程 Agent 实现

**关键设计问题**：
- 子 Agent 应该能看到父级的哪些上下文？
- 子 Agent 的 tool 权限是否受限？
- 如何防止子 Agent 无限递归创建孙 Agent？

---

### 3. 团队协议与通信机制 (Team Protocols)

**问题陈述**：
- 多个 Agent 如何协作完成单个任务？
- Agent 之间如何协商任务分配？
- 如何设计通用的 Agent 间通信协议？

**需要覆盖的技术点**：
- 智能体间通信协议设计（request-response 模式）
- 异步邮箱系统（async mailboxes）实现
- 委派任务的协商机制（接受/拒绝/协商）
- 团队角色定义与权限边界（leader/worker/specialist）
- 通信失败的重试与超时策略

**参考实现**：
- A2A (Agent-to-Agent) 协议
- MCP (Model Context Protocol) 的服务器发现机制
- 消息队列（Redis/RabbitMQ）用于 Agent 间通信

**关键设计问题**：
- 协议应该强类型还是松散结构？
- 如何处理 Agent 离线或响应超时？
- 是否需要中央协调者（orchestrator）？

---

### 4. 自治智能体与任务认领 (Autonomous Agents)

**问题陈述**：
- Agent 如何在没有用户输入的情况下自主运行？
- 如何让 Agent 自己发现和认领任务？
- 无中央协调者模式下如何分配工作？

**需要覆盖的技术点**：
- 任务看板（task board）扫描机制
- 智能体自主认领 vs 中央分配的策略对比
- 无 leader 模式的分布式决策（gossip/consensus）
- 心跳检测与存活监控
- 任务优先级动态调整

**参考实现**：
- IronClaw 的 Heartbeat 系统
- PicoClaw 的 cron 任务调度
- Edict 的事件驱动状态机

**关键设计问题**：
- 如何防止多个 Agent 争抢同一任务？
- 任务认领后的锁机制（分布式锁）
- Agent 故障时的任务重新分配

**需要覆盖的技术点**：

**现有覆盖**：tool-system-skills-analysis.md 有基础覆盖

**需要覆盖的技术点**：
- 慢操作异步化架构（长时间任务分离）

- 后台任务与主 Agent 的通信机制（callback/poll）
- 超时与取消机制（graceful shutdown）
- 后台任务结果持久化与查询

**现有覆盖**：workflow-task-engine-analysis.md 提及 IronClaw/NanoClaw 的后台任务

**需要补充内容**：
