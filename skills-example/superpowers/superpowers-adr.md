## ADR-001: Skill-Based Architecture

### 背景

早期 AI 编码助手的问题:
- 每次会话从零开始，无记忆
- 流程不一致，依赖提示词工程
- 无法系统化最佳实践
- 不同项目、不同用户经验差异大

### 决策

采用 **Skill-Based Architecture**:

```
┌─────────────────────────────────────┐
│          Skill (技能)                │
│  ┌─────────┐ ┌─────────┐ ┌────────┐ │
│  │ Trigger │ │  Flow   │ │ Output │ │
│  │ 触发条件 │ │ 执行流程 │ │  输出  │ │
│  └─────────┘ └─────────┘ └────────┘ │
└─────────────────────────────────────┘
```

**核心特征:**
1. 每个 Skill = 可复用的工作流单元
2. 自动触发（基于上下文）而非手动选择
3. 自包含（上下文 → 执行 → 结果）
4. 可组合（Skills 可相互调用）

### 考虑的替代方案

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| 单一系统提示 | 简单 | 无法扩展、难以维护 | ❌ 拒绝 |
| 函数调用 API | 程序化 | 平台依赖、不够灵活 | ❌ 拒绝 |
| Skill-Based | 模块化、可扩展、平台无关 | 初期设计成本高 | ✅ 接受 |

### 后果

**正面:**
- 系统化、可复用的最佳实践
- 跨项目、跨会话一致体验
- 社区可贡献 Skills
- 渐进式采用（先用基础 Skills，再深入）

**负面:**
- 需要学习 Skill 系统
- 前期设计投资大
- 技能发现机制必须完善

## ADR-002: Frontmatter Skill Definition

### 背景

需要一种方式定义 Skill:
- 机器可读（自动触发）
- 人类可读（文档）
- 版本控制友好
- 易于创建和维护

### 决策

使用 **YAML Frontmatter + Markdown**:

```markdown
---
name: skill-name
description: Use when [触发条件和症状]
---

# Skill Name

## Overview
...
```

**理由:**
1.  frontmatter 包含结构化元数据
2. Markdown 提供丰富文档
3. Git 友好（diff 可读）
4. 工具链成熟

### 考虑的替代方案

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| JSON Schema | 强类型 | 文档不友好 | ❌ 拒绝 |
| 纯代码 (Python/JS) | 灵活 | 门槛高、难以审计 | ❌ 拒绝 |
| XML | 结构化 | 冗长、不友好 | ❌ 拒绝 |
| YAML+Markdown | 平衡 | 无强类型检查 | ✅ 接受 |

### 后果

**正面:**
- 文档即代码
- 版本控制友好
- 社区贡献门槛低
- 可读性好

**负面:**
- 需要解析 frontmatter
- 无编译期类型检查
- 可能的格式不一致

## ADR-003: Multi-Platform Plugin Strategy

### 背景

AI 编码工具生态:
- Claude Code (Anthropic)
- Cursor
- Codex CLI (OpenAI)
- OpenCode
- Gemini CLI (Google)

每个平台有不同机制:
- 技能/提示系统
- 钩子/生命周期
- 配置格式

### 决策

**适配器模式 + 统一核心**

```
┌─────────────────────────────────────┐
│         统一 Skills 核心             │
│    (markdown skills, hooks)         │
├─────────────────────────────────────┤
│  .claude-plugin/ │ .cursor-plugin/  │
│  .codex/         │ .opencode/       │
│  └─ 平台特定适配                      │
└─────────────────────────────────────┘
```

**平台适配内容:**
- 入口点配置 (`CLAUDE.md`, `GEMINI.md`, `plugin.json`)
- 钩子注册方式
- 路径约定

**核心共享:**
- Skills 内容（完全复用）
- Hooks 逻辑（完全复用）
- 文档结构

### 考虑的替代方案

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| 单平台优先 | 深度集成 | 放弃其他平台 | ❌ 拒绝 |
| 完全独立 | 每平台优化 | 重复工作、不一致 | ❌ 拒绝 |
| 适配器模式 | 复用核心、灵活适配 | 需要维护适配层 | ✅ 接受 |

### 后果

**正面:**
- 一次编写，多平台运行
- 核心改进惠及所有平台
- 用户可切换工具保持流程

**负面:**
- 适配层维护成本
- 平台能力差异需处理
- 最低公分母问题

## ADR-004: Subagent-Driven Development

### 背景

单个会话限制:
- 上下文窗口有限
- 长时间会话性能下降
- 无法真正并行工作

### 决策

**子代理作为主要执行机制**

```
主代理
  ├── 调度子代理 A (任务 1)
  ├── 调度子代理 B (任务 2)
  └── 调度子代理 C (评审)
```

**执行模式:**
1. 主代理规划
2. 子代理执行独立任务
3. 每个子代理有精确上下文（零继承）
4. 主代理整合结果

### 考虑的替代方案

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| 单会话执行 | 简单 | 无法扩展、易出错 | ❌ 拒绝 |
| 手动多会话 | 灵活 | 不一致、用户负担 | ❌ 拒绝 |
| 子代理自动调度 | 可扩展、系统化 | 需要编排逻辑 | ✅ 接受 |

### 后果

**正面:**
- 突破上下文限制
- 真正的并行处理
- 任务隔离减少错误传播
- 可扩展（更多子代理）

**负面:**
- 子代理调度开销
- 需要上下文管理
- 调试更复杂
- 平台支持不一致（Codex 支持，其他可能不支持）

## ADR-005: Two-Phase Code Review

### 背景

代码评审的常见问题:
- 遗漏功能偏差（实现偏离需求）
- 只关注代码风格，忽略正确性
- 评审太晚，返工成本高

### 决策

**强制两阶段评审:**

```
Phase 1: Spec Compliance (规范合规)
  ├── 是否按要求实现？
  ├── 是否有功能偏差？
  └── IF 不合规 → 返回修改

Phase 2: Code Quality (代码质量)
  ├── 代码质量评估
  ├── 识别问题
  └── IF 有关键问题 → 修复
```

**关键规则:**
- 阶段1必须先通过，才能进入阶段2
- 阶段1由 spec-document-reviewer 执行
- 阶段2由 code-reviewer 执行
- 两个评审使用不同上下文（避免偏见）

### 考虑的替代方案

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| 单阶段评审 | 快速 | 容易遗漏功能问题 | ❌ 拒绝 |
| 人工评审 | 高质量 | 不可扩展、延迟 | ❌ 拒绝 |
| 无评审 | 最快 | 质量失控 | ❌ 拒绝 |
| 两阶段自动评审 | 平衡 | 需要编排 | ✅ 接受 |

### 后果

**正面:**
- 功能偏差早期捕获
- 关注点分离（合规 vs 质量）
- 系统化、可复现

**负面:**
- 评审开销翻倍
- 需要维护两个评审代理
- 可能过于严格

## ADR-006: Git Worktree Isolation

### 背景

开发环境问题:
- 多任务切换时状态污染
- 实验性更改影响主线
- 难以同时进行多个功能开发

### 决策

**强制 Git Worktree 隔离**

```
main-repo/
├── .git/           ← 共享仓库
├── src/            ← main branch
└── .worktrees/
    ├── feature-a/  ← feature/a branch
    └── feature-b/  ← feature/b branch
```

**触发条件:**
- 开始新功能开发 → 自动创建 worktree
- 复杂修复 → 创建 worktree
- 任何非 trivial 更改

**安全检查:**
- `.worktrees/` 必须在 `.gitignore`
- 创建前验证基线测试通过

### 考虑的替代方案

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| 单工作区 | 简单 | 状态污染 | ❌ 拒绝 |
| 手动分支切换 | 原生 Git | 易出错、上下文丢失 | ❌ 拒绝 |
| Stash/暂存 | 轻量 | 容易遗忘 | ❌ 拒绝 |
| Git Worktree | 真正隔离 | 磁盘占用 | ✅ 接受 |

### 后果

**正面:**
- 完全隔离的工作区
- 并行开发多个功能
- 安全的实验环境
- 快速切换（无需 stash/pop）

**负面:**
- 磁盘空间占用
- 需要理解 worktree 概念
- 依赖管理可能重复（npm install）

## ADR-007: Strict TDD Enforcement

### 背景

TDD 的常见失败:
- "先写代码，后补测试"
- "这次情况特殊"
- "我已经手动测试过了"
- 代码先于测试成为习惯

### 决策

**铁律: NO CODE WITHOUT A FAILING TEST FIRST**

```
RED: 写测试 → 运行 → 确认失败
  ↓
GREEN: 写最小代码让测试通过
  ↓
REFACTOR: 清理代码，保持测试通过
  ↓
[循环]
```

**违规处理:**
- 代码在测试前写？→ 删除它
- 没有例外
- 不允许保留作为"参考"
- 必须重新开始

**强制执行:**
- `test-driven-development` Skill 内化规则
- 代理自我监控
- `verification-before-completion` 验证红绿循环

### 考虑的替代方案

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| 推荐 TDD | 灵活 | 不执行等于没有 | ❌ 拒绝 |
| 后补测试 | 快速开始 | 测试质量差、遗漏 | ❌ 拒绝 |
| 无测试 | 最快 | 质量失控 | ❌ 拒绝 |
| 铁律强制 | 质量保证 | 学习曲线、初期慢 | ✅ 接受 |

### 后果

**正面:**
- 设计驱动（测试即设计）
- 高覆盖率
- 可重构的代码
- 回归防护

**负面:**
- 初期开发速度较慢
- 学习曲线陡峭
- 需要纪律维持
- 可能过度测试

## ADR-008: Verification Gates

### 背景

虚假完成的常见问题:
- "应该可以通过了"
- "我看看代码，应该没问题"
- "之前运行过测试"
- 声称完成但未验证

### 决策

**强制验证门 (Gate Function)**

```
BEFORE 任何状态声称:

1. IDENTIFY: 什么命令能证明这个声称？
2. RUN: 执行完整命令（新鲜、完整）
3. READ: 完整输出，检查退出码，计数失败
4. VERIFY: 输出是否确认声称？
5. ONLY THEN: 做出声称
```

**关键规则:**
- 未运行验证命令 = 不能声称
- 使用 "should", "probably" = 红旗
- 满意表达前必须验证
- 验证证据必须新鲜

### 考虑的替代方案

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| 信任开发者 | 高效 | 虚假完成、信任破裂 | ❌ 拒绝 |
| 事后验证 | 快速迭代 | 问题发现晚 | ❌ 拒绝 |
| 持续集成 | 自动化 | 反馈延迟 | ⚪ 补充 |
| 强制验证门 | 即时 | 打断流程 | ✅ 接受 |

### 后果

**正面:**
- 诚实、可验证的进度
- 问题早期发现
- 建立信任

**负面:**
- 频繁运行命令
- 可能打断心流
- 需要纪律

## ADR-009: Parallel Agent Dispatch

### 背景

调试多失败时的痛点:
- 逐个调查，时间线性增长
- 失败可能相关，难以判断
- 串行处理浪费等待时间

### 决策

**并行代理调度**

```
3 个独立失败
    │
    ├──→ 代理 A 调查失败 1 ──┐
    ├──→ 代理 B 调查失败 2 ──┼──→ 整合结果
    └──→ 代理 C 调查失败 3 ──┘

(同时执行，非串行)
```

**使用条件:**
- 3+ 失败跨多个文件
- 失败相互独立
- 无需共享上下文理解
- 代理不会互相干扰

**提示结构:**
1. Focused - 一个清晰问题域
2. Self-contained - 所有需要上下文
3. Specific output - 代理应返回什么

### 考虑的替代方案

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| 串行调查 | 简单 | 慢 | ❌ 拒绝 |
| 单代理全面分析 | 上下文完整 | 可能超出上下文窗口 | ⚪ 备选 |
| 并行代理 | 快速 | 需要独立问题 | ✅ 接受 |

### 后果

**正面:**
- 调查时间从 O(n) 到 O(1)
- 独立问题隔离
- 并发利用

**负面:**
- 需要平台支持多代理
- 复杂问题难以并行
- 结果整合开销

## ADR-010: Skill Creation TDD

### 背景

编写 Skills 的挑战:
- 难以验证 Skill 有效性
- 创建后发现问题成本高
- 没有系统化测试方法

### 决策

**Skill TDD: 文档的测试驱动开发**

```
TDD 概念 → Skill 创建映射:

测试用例    → 子代理压力场景
生产代码   → Skill 文档 (SKILL.md)
测试失败   → 无技能时代理违规
测试通过   → 有技能时代理遵守
重构       → 修复漏洞同时保持合规
```

**流程:**
1. 识别 Skill 要解决的问题
2. 创建压力场景（子代理会违规的场景）
3. 运行场景，确认违规（RED）
4. 编写 Skill 文档（GREEN）
5. 重新运行场景，确认合规
6. 重构优化（REFACTOR）

**铁律:** `NO SKILL WITHOUT A FAILING TEST FIRST`

### 考虑的替代方案

| 方案 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| 直接编写 | 快速 | 质量问题 | ❌ 拒绝 |
| 人工评审 | 高质量 | 不可扩展 | ⚪ 补充 |
| Skill TDD | 系统化 | 需要投资 | ✅ 接受 |

### 后果

**正面:**
- Skill 质量可验证
- 问题早期发现
- 系统化改进流程
- 与开发 Skills 一致的体验

**负面:**
- 需要设计压力场景
- 测试基础设施成本
- 初期开发时间增加
