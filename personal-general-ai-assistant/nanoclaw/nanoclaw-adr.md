# NanoClaw 架构决策记录 (ADR)

> 关键设计决策及其理由

---

## ADR-001: 容器隔离优于应用层权限

### 背景

设计一个执行 AI 代理任务的多通道消息系统时，必须决定如何隔离不同用户群组和保护主机系统。

**考虑的选项**:
1. **应用层权限检查**: 在代码中实现复杂的权限系统，检查每个操作是否允许
2. **容器隔离**: 将每个群组的代理运行在独立容器中，通过操作系统级隔离保护
3. **进程级隔离**: 使用单独的进程但共享文件系统
4. **虚拟机隔离**: 为每个群组启动完整 VM

### 决策

**采用容器隔离**（Apple Container Framework / Docker）。

### 后果

**积极**:
- 安全边界由操作系统/ hypervisor 保证，非应用逻辑
- 文件系统隔离是隐含的，无需代码实现
- 网络隔离防止容器直接访问外部 API
- 资源限制（CPU、内存）可由容器运行时强制执行

**消极**:
- 增加容器运行时依赖
- 需要处理容器生命周期管理（启动、停止、超时）
- 容器内文件操作需要挂载配置

### 权衡分析

| 方案 | 安全强度 | 复杂度 | 性能 | 适用性 |
|------|----------|--------|------|--------|
| 应用层权限 | 低 | 高 | 高 | ❌ |
| 容器隔离 | 高 | 中 | 中 | ✅ |
| 进程级隔离 | 中 | 中 | 高 | ❌ |
| VM 隔离 | 极高 | 高 | 低 | ❌ |

> **Ousterhout 原则**: Security Through Isolation - 安全应通过隔离实现，而非应用层权限检查。

## ADR-002: 文件系统 IPC 优于网络套接字

### 背景

主机与容器内的代理需要双向通信。必须选择通信机制。

**考虑的选项**:
1. **Unix Domain Socket**: 传统 IPC 机制
2. **TCP/UDP Socket**: 网络套接字
3. **文件系统 IPC**: 通过共享目录的读写文件通信
4. **stdin/stdout 管道**: 标准流通信
5. **HTTP API**: 容器内运行 HTTP 服务

### 决策

**采用文件系统 IPC**（`data/ipc/{group}/input/`, `output/`, `messages/`, `tasks/`）。

### 后果

**积极**:
- 无网络配置依赖
- 持久化（崩溃后可恢复未处理消息）
- 简单调试（直接查看文件内容）
- 容器无需网络访问即可通信
- 支持流式输出（边写边读）

**消极**:
- 需要文件系统轮询
- 需要清理过期文件
- 延迟略高于内存通信

### 权衡分析

```
持久化 vs 性能
├── Socket: 高性能，非持久
├── HTTP:   中性能，需网络配置
├── 文件:    中性能，天然持久 ✅
└── 管道:    低延迟，单向受限
```

> **关键考量**: 崩溃恢复能力优先于极致性能。消息不会丢失比消息快 10ms 到达更重要。

## ADR-003: SQLite 优于其他数据库

### 背景

需要持久化存储消息、任务、会话等数据。必须选择存储方案。

**考虑的选项**:
1. **JSON 文件**: 简单，无需依赖
2. **SQLite**: 嵌入式关系数据库
3. **PostgreSQL/MySQL**: 客户端-服务器数据库
4. **Redis**: 内存数据库
5. **LevelDB/RocksDB**: 嵌入式 KV 存储

### 决策

**采用 SQLite**（通过 `better-sqlite3`）。

### 后果

**积极**:
- 零配置（无需服务器进程）
- 事务支持保证数据一致性
- SQL 查询能力（复杂查询、索引、聚合）
- 单文件存储便于备份和迁移
- 成熟稳定，广泛测试

**消极**:
- 原生模块编译依赖
- 高并发写入性能受限（单文件锁）
- 不如专用 KV 存储轻量

### 权衡分析

| 方案 | 配置复杂度 | 查询能力 | 事务 | 体积 | 选择 |
|------|------------|----------|------|------|------|
| JSON | 无 | 差 | 无 | 小 | ❌ |
| SQLite | 无 | 强 | 有 | 中 | ✅ |
| PG/MySQL | 高 | 强 | 有 | 大 | ❌ |
| Redis | 中 | 中 | 无 | 中 | ❌ |
| LevelDB | 无 | 差 | 无 | 小 | ❌ |

> **Ousterhout 原则**: 深模块 - SQLite 提供简单接口（SQL），隐藏复杂实现（B-tree、查询优化、ACID）。

## ADR-004: 直接 SQL 优于 ORM

### 背景

使用 SQLite 后，必须决定如何与数据库交互。

**考虑的选项**:
1. **ORM (Prisma/TypeORM/Drizzle)**: 对象关系映射
2. **Query Builder (Knex)**: 链式查询构造器
3. **直接 SQL**: 手写 SQL 语句

### 决策

**采用直接 SQL**（`better-sqlite3` 的 `prepare().run()` API）。

### 后果

**积极**:
- 无抽象泄漏（调用者完全理解底层操作）
- 零 ORM 依赖体积
- 精确控制查询（索引使用、执行计划）
- 无生成的代码/类型定义
- Schema 变更简单直接

**消极**:
- 手动管理 Schema 迁移
- 无自动类型安全（需配合 Zod）
- SQL 知识要求

### 代码对比

```typescript
// ORM 方式 (TypeORM)
const messages = await Message.find({
  where: { chatJid, timestamp: { $gt: since } },
  order: { timestamp: 'ASC' }
});

// 直接 SQL 方式
const stmt = db.prepare(`
  SELECT * FROM messages
  WHERE chat_jid = ? AND timestamp > ?
  ORDER BY timestamp ASC
`);
const messages = stmt.all(chatJid, since);
```

> **Ousterhout 原则**: 避免抽象泄漏。ORM 隐藏了 SQL，但泄漏了映射复杂性。

## ADR-005: 自注册优于配置

### 背景

频道系统需要支持多通道（WhatsApp、Telegram、Discord 等）。必须决定如何发现和加载频道实现。

**考虑的选项**:
1. **配置文件**: JSON/YAML 配置指定启用哪些频道
2. **自动扫描**: 扫描目录动态加载模块
3. **显式导入**: 主文件显式导入所有频道
4. **自注册**: 每个频道模块启动时注册自己

### 决策

**采用自注册模式**（`registerChannel(name, factory)`）。

### 后果

**积极**:
- 零配置（无需维护配置文件）
- 插件化扩展（新频道只需添加文件）
- 延迟加载（工厂返回 null 时自动跳过）
- 类型安全（编译时检查）

**消极**:
- 隐式依赖（难以追踪哪些频道已注册）
- 启动顺序依赖

### 实现模式

```typescript
// channels/whatsapp.ts
import { registerChannel } from './registry.js';

registerChannel('whatsapp', (opts) => {
  if (!process.env.WHATSAPP_TOKEN) return null; // 未配置，跳过
  return new WhatsAppChannel(opts);
});

// registry.ts - 核心实现仅 29 行
const factories = new Map<string, ChannelFactory>();

export function registerChannel(name: string, factory: ChannelFactory) {
  factories.set(name, factory);
}
```

> **设计模式**: 工厂模式 + 副作用注册。简单但有效。

## ADR-006: Git 分支技能优于插件架构

### 背景

需要扩展功能（添加新频道、工具等）。必须决定扩展机制。

**考虑的选项**:
1. **插件系统**: 动态加载 `.so`/`.dll`/`.js` 插件
2. **NPM 包**: 发布和安装扩展包
3. **Git 子模块**: 引用外部代码
4. **Git 分支**: 功能分支，通过合并添加
5. **配置脚本**: 运行时下载和执行脚本

### 决策

**采用 Git 分支技能系统**（`/add-whatsapp`, `/add-telegram` 等分支）。

### 后果

**积极**:
- 版本控制（每个技能有完整历史）
- 原子性（合并一个提交获得完整功能）
- 可审查（PR/MR 流程）
- 可组合（分支可基于其他分支）
- 无运行时动态代码执行风险

**消极**:
- 需要 Git 知识
- 合并冲突需要手动解决
- 技能卸载需 revert

### 工作流

```bash
# 添加 WhatsApp 支持
git fetch origin add-whatsapp
git merge origin/add-whatsapp --no-edit

# 技能包含：
# - src/channels/whatsapp.ts
# - container/skills/whatsapp/
# - package.json 依赖更新
```

> **哲学**: "代码即配置" - 功能通过代码修改而非配置项启用。

## ADR-007: 凭证代理优于环境变量传递

### 背景

容器内的代理需要调用 Anthropic API，但不能直接暴露 API key。

**考虑的选项**:
1. **环境变量传递**: 将 API key 作为环境变量传入容器
2. **文件挂载**: 挂载包含密钥的文件
3. **凭证代理**: 主机运行代理，容器通过代理访问 API
4. **短期令牌**: 生成临时令牌传入容器

### 决策

**采用凭证代理**（HTTP 代理，主机注入密钥）。

### 后果

**积极**:
- 容器从不持有真实密钥（仅知道代理地址）
- 可审计（代理记录所有请求）
- 可限流（代理控制请求速率）
- 支持 OAuth（令牌刷新在主机处理）

**消极**:
- 增加网络跳数（轻微延迟）
- 需要管理代理生命周期
- 容器必须配置代理地址

### 安全模型

```
┌─────────────┐      ┌─────────────┐      ┌─────────────────┐
│  Container  │ ───▶ │   Proxy     │ ───▶ │  Anthropic API  │
│  (no key)   │      │ (injects)   │      │                 │
└─────────────┘      └─────────────┘      └─────────────────┘
                            │
                            ▼
                     ┌─────────────┐
                     │   Env File  │
                     │ (host only) │
                     └─────────────┘
```

## ADR-008: 按群组容器隔离

### 背景

多个群组共享一个 NanoClaw 实例。必须决定隔离粒度。

**考虑的选项**:
1. **单容器共享**: 所有群组共享一个容器
2. **按频道容器**: 每个消息通道一个容器
3. **按群组容器**: 每个聊天群组一个容器
4. **按用户容器**: 每个发送者一个容器
5. **按任务容器**: 每个任务一个容器

### 决策

**采用按群组容器隔离**（`groups/{folder}/` → 独立容器）。

### 后果

**积极**:
- 群组间完全隔离（文件系统、会话、历史）
- 一个群组崩溃不影响其他群组
- 资源使用可独立监控和限制
- 符合用户心理模型（群组是独立工作区）

**消极**:
- 更多容器开销（内存、启动时间）
- 需要管理更多容器生命周期
- 全局并发限制更复杂

### 隔离级别对比

| 粒度 | 隔离强度 | 资源效率 | 用户体验 | 选择 |
|------|----------|----------|----------|------|
| 单容器共享 | 无 | 极高 | 差 | ❌ |
| 按频道 | 低 | 高 | 中 | ❌ |
| 按群组 | 高 | 中 | 高 | ✅ |
| 按用户 | 极高 | 低 | 中 | ❌ |
| 按任务 | 极高 | 极低 | 差 | ❌ |

## ADR-009: 最小依赖原则

### 背景

Node.js 生态有依赖膨胀问题。必须决定依赖策略。

**考虑的选项**:
1. **标准方案**: 使用 Express、Lodash、Moment、Axios 等常用库
2. **框架方案**: 使用 NestJS/Fastify 等框架
3. **最小方案**: 仅使用不可替代的依赖
4. **自研方案**: 自己实现大部分功能

### 决策

**采用最小依赖策略**（仅 6 个核心生产依赖）。

### 依赖清单

| 包名 | 用途 | 为何不可替换 |
|------|------|--------------|
| `better-sqlite3` | SQLite 绑定 | 原生性能，无纯 JS 替代 |
| `cron-parser` | Cron 解析 | 表达式解析复杂，需可靠库 |
| `pino` | 日志 | 结构化日志，性能最优 |
| `pino-pretty` | 日志美化 | 开发体验 |
| `yaml` | YAML 解析 | 技能配置需要 |
| `zod` | 类型校验 | 运行时安全 |

### 主动避免的依赖

| 常见依赖 | 为何不用 | 替代方案 |
|----------|----------|----------|
| Express/Fastify | 无需 HTTP 服务 | IPC 通信 |
| Lodash | 现代 JS 已足够 | 原生方法 |
| Moment.js | 体积大 | 原生 `Date` |
| Axios | Node 18+ 有 fetch | 原生 `fetch` |
| ORM | 抽象泄漏 | 直接 SQL |

> **结果**: 生产依赖体积 ~6MB（对比典型 Node 项目 100MB+）

## ADR-010: 代码即配置

### 背景

传统应用通过配置文件定制行为。需要决定配置策略。

**考虑的选项**:
1. **配置文件**: JSON/YAML/TOML 配置
2. **环境变量**: 12-Factor 应用风格
3. **数据库配置**: 配置存储在 DB 中
4. **代码配置**: 通过修改代码定制

### 决策

**采用代码即配置**（修改 `src/config.ts`，而非外部配置）。

### 后果

**积极**:
- 类型安全（编译时检查）
- IDE 支持（自动完成、重构）
- 版本控制（配置变更可审查）
- 无配置解析代码
- 无配置验证逻辑

**消极**:
- 非技术用户无法配置
- 升级需处理代码冲突
- 重新部署才能变更配置

### 适用场景

| 场景 | 代码配置 | 外部配置 |
|------|----------|----------|
| 开发者工具 | ✅ | ❌ |
| 最终用户应用 | ❌ | ✅ |
| 个人项目 | ✅ | ❌ |
| 企业软件 | ❌ | ✅ |

> **适用性**: NanoClaw 是个人开发者工具，代码配置更合适。

### 配置示例

```typescript
// src/config.ts
export const ASSISTANT_NAME = 'NanoClaw';
export const POLL_INTERVAL = 3000;
export const MAX_CONCURRENT_CONTAINERS = 5;
export const CONTAINER_TIMEOUT = 120_000;

// 类型安全，IDE 支持，重构友好
```

## ADR-011: 双游标消息追踪

### 背景

需要可靠的消息处理，确保不遗漏、不重复。

**考虑的选项**:
1. **单游标**: 一个全局位置标记
2. **双游标**: 已见游标 + 已处理游标
3. **ACK 确认**: 每条消息确认后才标记完成
4. **幂等处理**: 允许重复处理，结果相同

### 决策

**采用双游标机制**（`lastTimestamp` + `lastAgentTimestamp`）。

### 机制说明

```
消息流:  [A] [B] [C] [D] [E]
         ↑               ↑
    lastTimestamp    lastAgentTimestamp
    (全局已见)        (已处理)

崩溃恢复: 对比两个游标，识别未处理消息 [D] [E]
```

### 后果

**积极**:
- 崩溃后可识别未处理消息
- 重试不重复发送（回滚处理游标）
- 无需每条消息 ACK

**消极**:
- 增加状态复杂度
- 可能重复处理（可接受）

### 状态机

```
PENDING → RUNNING → SUCCESS
                   ↓
                   ERROR → RETRYING → PENDING
```

## ADR-012: 指数退避重试

### 背景

容器执行可能失败（网络超时、API 限流）。需要决定重试策略。

**考虑的选项**:
1. **立即重试**: 失败后立即重试
2. **固定间隔**: 固定延迟后重试
3. **线性退避**: 延迟随重试次数线性增加
4. **指数退避**: 延迟随重试次数指数增加
5. **熔断**: 失败多次后停止服务

### 决策

**采用指数退避重试**（`delay = 1000 * 2^(retry-1)`，最大 5 次）。

### 计算公式

```typescript
const BASE_RETRY_MS = 1000;
const MAX_RETRIES = 5;

function getRetryDelay(retryCount: number): number {
  return BASE_RETRY_MS * Math.pow(2, retryCount - 1);
}

// 结果: 1000ms, 2000ms, 4000ms, 8000ms, 16000ms
```

### 后果

**积极**:
- 快速失败时的快速恢复
- 持续失败时减少负载（给对方恢复时间）
- 避免雪崩效应

**消极**:
- 严重故障时恢复延迟较长
- 需要跟踪重试次数状态
